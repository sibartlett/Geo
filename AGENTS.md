## Project overview

**Geo** is a geospatial library for .NET, published on NuGet as [`Geo`](https://nuget.org/packages/Geo).
It provides geographic geometry types, GPS data types, serializers for common
spatial and GPS file formats, and geodetic/geomagnetic calculations. It is
designed specifically for geographic data (as opposed to planar/projected data).

Conventions used throughout the library:

- All ordinates are in **degrees** unless otherwise specified.
- All measurements are in **S.I. units** (metres, seconds, etc.) unless otherwise specified.
- The coordinate reference system is assumed to be **WGS-84**.
- Ordinate order in constructors is **latitude, longitude** (e.g. `new Coordinate(lat, lon)`).

### `Geo/` source structure

| Directory | Contents |
|-----------|----------|
| `Geo/` (root) | Core types: `Coordinate`, `CoordinateZ/M/ZM`, `CoordinateSequence`, `Envelope`, `LineSegment`, `GeoContext`, `Constants`, `SimpleJson` (bundled JSON parser), `SpatialEqualityOptions` |
| `Abstractions/` | Base classes (`SpatialObject`, `Geometry`, `SpatialReadOnlyCollection`) and `Interfaces/` (`IGeometry`, `IPosition`, `IHasArea`, `IHasLength`, `ISpatialEquatable`, `Is3D`, `IsMeasured`, `IMeasure`, `IGeoJsonObject`, `IGeodeticCalculator`) |
| `Geometries/` | `Point`, `LineString`, `LinearRing`, `Polygon`, `Triangle`, `Circle`, `MultiPoint`, `MultiLineString`, `MultiPolygon`, `GeometryCollection` |
| `Geodesy/` | Distance/bearing calculations: `SphereCalculator`, `SpheroidCalculator`, `Spheroid`, `GeodeticLine`, `GeodeticCalculations` |
| `Geomagnetism/` | Declination/inclination/intensity via IGRF and WMM models; `Models/` holds `Wmm1985`–`Wmm2025` and `IgrfModelFactory` |
| `Gps/` | `GpsData`, `Route`, `Track`, `TrackSegment`, `Waypoint`, `GpsFeatures`; `Metadata/`; `Serialization/` for GPX/NMEA/IGC/flightplan readers |
| `IO/` | Serializers: `Wkt/` (Well-Known Text), `Wkb/` (Well-Known Binary), `GeoJson/`, `Google/` (polyline encoding) |
| `Linq/` | `EnumerableExtensions`, spatial comparers |
| `Measure/` | Unit types: `Distance`, `Area`, `Speed` with their `*Unit` enums and extension methods |

## Build & test

The project uses [NUKE](https://nuke.build/) for build automation. **Prefer the
bootstrap scripts** over calling `dotnet` directly, since they pin the SDK and
run the same targets as CI:

```bash
./build.sh                          # default target: Compile
./build.sh Test                     # build + run tests
./build.sh CheckForUncommittedChanges Test   # exactly what CI runs
./build.sh Clean
./build.sh Publish                  # dotnet publish in Release
```

On Windows use `build.cmd` / `build.ps1` (same targets).

NUKE targets are defined in `build/Build.cs`:

- `Restore` → `Compile` → `Test` (each depends on the previous)
- `Clean` runs before `Restore`
- `CheckForUncommittedChanges` runs `dotnet husky run --group verify`, which
  runs `csharpier check .` — it fails if any file is not formatted.
- `Publish` produces the Release package.

You can also use plain dotnet if the SDK is already present:

```bash
dotnet build Geo.sln
dotnet test Geo.Tests/Geo.Tests.csproj
```

Note: building the `Geo` project triggers an MSBuild `Husky` target that runs
`dotnet tool restore` and `dotnet husky install` (guarded by the `HUSKY` MSBuild
property — set `HUSKY=0` to skip). The `Geo` package is generated on build
(`GeneratePackageOnBuild=true`).

## Code style & formatting

**Formatting is enforced by [CSharpier](https://csharpier.com/) and is a CI gate.**
Before committing, run:

```bash
dotnet csharpier format .     # format the whole repo
dotnet csharpier check .      # verify formatting (what CI runs)
```

A Husky.Net **pre-commit hook** (`.husky/task-runner.json`) automatically runs
`csharpier format` on staged files. Both csharpier and husky are pinned as local
dotnet tools in `.config/dotnet-tools.json` (run `dotnet tool restore` first if needed).

C# conventions observed in the codebase (see `build/.editorconfig`):

- File-scoped namespaces (`namespace Geo.Geometries;`).
- No `this.` qualification; no explicit accessibility modifiers where the default suffices.
- Expression-bodied members are preferred for properties/accessors/indexers.
- Target framework of the library is **netstandard2.0** — keep new code compatible
  with that (avoid newer BCL APIs not available there). Tests and build target net8.0.
- Geometry types implement value equality plus a spatial-equality overload that
  takes `SpatialEqualityOptions` (see `Geometries/Point.cs` for the pattern:
  `Equals`/`GetHashCode`, `Equals(object, SpatialEqualityOptions)`, `==`/`!=`).

## Testing conventions

- Framework: **xUnit** (`[Fact]` / `[Theory]`), with `coverlet.collector` for coverage.
- Tests mirror the source namespace/folder layout under `Geo.Tests/`.
- Test method names use `Snake_case_describing_the_behaviour`
  (e.g. `Null_input_string_throws_argument_exception`).
- Serializer tests read sample files from the top-level `reference/` directory
  (see `Gps/Serialization/SerializerTestFixtureBase.cs`). Add new sample data there.

## Key architectural notes

- **`GeoContext.Current`** is an ambient, mutable settings object holding the active
  `Spheroid`, geodetic calculator, geomagnetism calculator, `SpatialEqualityOptions`,
  and `LongitudeWrapping` flag. Calculations default to reading from it.
- **Coordinate hierarchy**: `Coordinate` (lat/lon) is the base; `CoordinateZ` adds
  elevation, `CoordinateM` adds a measure, `CoordinateZM` adds both. Geometries expose
  `Is3D` / `IsMeasured` reflecting which coordinate variant they hold.
- **Serialization** is split into reader/writer pairs, each with a `*Settings` class
  (e.g. `WktReader`/`WktWriter`/`WktWriterSettings`, `WkbReader`/`WkbWriter`,
  `GeoJsonReader`/`GeoJsonWriter`). WKT tokenizing lives in `IO/Wkt/Wkt*Token*`.
- **GPS deserializers** implement `IGpsFileDeSerializer`; several formats are
  deserialize-only (NMEA, IGC, Garmin/PocketFMS/SkyDemon flightplans), while GPX
  supports both read and write (versions 1.0 and 1.1).
- **Geomagnetism models** are code-generated coefficient tables. When a new WMM/IGRF
  epoch is released, add a new `Models/WmmYYYY.cs` (or update `IgrfModelFactory`) —
  follow the existing files exactly and bump the version accordingly.
- `SimpleJson.cs` is a vendored JSON implementation (guarded by the
  `SIMPLE_JSON_TYPEINFO` compile constant) so the library has no external JSON dependency.

## Versioning & releases

- The package version lives in `Geo/Geo.csproj` (`<Version>`). Bumping it is a
  deliberate, standalone commit (see history: "Bump version to x.y.z").
- CI (`.github/workflows/ci.yml`) runs on every push: `CheckForUncommittedChanges Test`
  on ubuntu-latest. The workflow is generated from the `[GitHubActions]` attribute in
  `build/Build.cs` — regenerate with `nuke --generate-configuration GitHubActions_ci`
  rather than editing the YAML directly.

## When making changes

1. Keep new library code **netstandard2.0-compatible**.
2. Run `dotnet csharpier format .` before committing (or rely on the pre-commit hook).
3. Add/adjust xUnit tests under the mirrored path in `Geo.Tests/`.
4. Run `./build.sh CheckForUncommittedChanges Test` locally to reproduce CI.
5. Only bump `<Version>` when intentionally cutting a release.
